# Преобразователь регулярной грамматики в конечный автомат

## Описание проекта

Программа принимает на вход текст регулярной грамматики (праволинейной или леволинейной) и строит соответствующий конечный автомат. Если полученный НКА оказывается недетерминированным, выполняется его детерминизация. Результат выводится в виде списка рёбер графа в текстовом формате.

## Логика работы

### Шаг 1: Ввод грамматики
**Функция:** `main` (модуль `Main.hs`)

- Читает входную строку из аргументов командной строки (имя файла) или, если аргументов нет, из `stdin`.
- Удаляет символы перевода строки.

### Шаг 2: Разбор
**Функция:** `parseGrammar` (модуль `Grammar.hs`)

- Принимает строку вида `"H=aN|bN;N=cN|d"`.
- Разбивает по `;` на отдельные правила.
- Каждое правило разбивает по `=` на левую (нетерминал) и правую части.
- Правую часть разбивает по `|` на альтернативы.
- Каждую альтернативу преобразует в тип `Alternative` (`TerminalOnly`, `RightLinearAlt` или `LeftLinearAlt`).
- Возвращает значение типа `Grammar`.

### Шаг 3: Определение типа грамматики
**Функция:** `getGrammarType` (модуль `Grammar.hs`)

- Анализирует все альтернативы во всех правилах.
- Если встречаются `RightLinearAlt` — тип `RightLinear`.
- Если встречаются `LeftLinearAlt` — тип `LeftLinear`.
- Если только `TerminalOnly` — по умолчанию `RightLinear`.
- При смешанных типах выбрасывается ошибка.

### Шаг 4: Валидация
**Функция:** `validateGrammar` (модуль `Grammar.hs`)

- Проверяет, что все нетерминалы в правых частях определены в левых частях.
- Убеждается, что все альтернативы соответствуют заявленному типу грамматики.
- Проверяет, что пустые альтернативы (отсутствие правой части) встречаются только в леволинейных грамматиках.

### Шаг 5: Построение НКА
**Функция:** `grammarToNFA` (модуль `Convert.hs`)

**Для праволинейной грамматики:**
- Начальное состояние — `H`, добавляется финальное состояние `#`.
- Правило `A = a` → переход `A -a-> #`.
- Правило `A = aB` → переход `A -a-> B`.

**Для леволинейной грамматики:**
- Добавляется начальное состояние `@`.
- Финальные состояния — нетерминалы с пустыми альтернативами (правила без правой части).
- Правило `A = a` → переход `@ -a-> A`.
- Правило `A = Ba` → переход `B -a-> A`.

### Шаг 6: Проверка детерминированности
**Функция:** `isDeterministic` (модуль `Determinize.hs`)

- Проходит по всем переходам НКА.
- Если для какой-либо пары `(состояние, символ)` множество целевых состояний содержит больше одного элемента, возвращает `False`.
- Иначе — `True`.

### Шаг 7: Детерминизация
**Функция:** `nfaToDFA` (модуль `Determinize.hs`)

- Начальное состояние ДКА — множество `{ начальное_состояние_НКА }`.
- Для каждого необработанного множества состояний и каждого символа алфавита вычисляется объединение переходов.
- Новые множества добавляются как состояния ДКА.
- Имя состояния ДКА формируется как строка: одиночное состояние сохраняет исходное имя, множества записываются в фигурных скобках через запятую (например, `{A,B}`).
- Финальные состояния ДКА — множества, содержащие хотя бы одно финальное состояние НКА.

### Шаг 8: Вывод результата
**Функции:** `dfaToEdges`, `edgesToString` (модуль `Automaton.hs`)

- `dfaToEdges` преобразует ДКА в список рёбер `(from, sym, to)`.
- `edgesToString` форматирует каждое ребро как `"from -> sym -> to"` и объединяет переводом строки.
- `main` выводит результат в требуемом формате.

## Пример вывода

```
################################################################################
[1] Входная грамматика:
    "H=aA|aB;A=b;B=c"

[2] Результат парсинга:
    Тип: RightLinear
    Начальный символ: H
    Продукции:
        A -> b
        B -> c
        H -> aB | aA

[3] Недетерминированный конечный автомат (НКА):
    Начальное состояние: H
    Финальные состояния: {#}
    Переходы:
        A -b-> {#}
        B -c-> {#}
        H -a-> {A,B}

[4] Проверка детерминированности: НЕТ (требуется детерминизация)

[5] Детерминированный конечный автомат (ДКА):
    Начальное состояние: H
    Финальные состояния:
        #
    Переходы:
        {A,B} -b-> #
        {A,B} -c-> #
        H -a-> {A,B}

[6] Результат (список рёбер):
{A,B} -> b -> #
{A,B} -> c -> #
H -> a -> {A,B}
################################################################################
```

## Тестирование

Набор команд для проверки всех основных случаев (PowerShell):

```powershell
# 1. Праволинейная детерминированная
echo "H=aA;A=bB;B=c" | stack exec regexp-to-automaton-exe

# 2. Праволинейная недетерминированная
echo "H=aA|aB;A=b;B=c" | stack exec regexp-to-automaton-exe

# 3. Праволинейная с циклом
echo "H=aH|bH|c" | stack exec regexp-to-automaton-exe

# 4. Леволинейная с пустыми правилами
echo "S=Aa|Bb;A=Cc;B=Dd;C=;D=" | stack exec regexp-to-automaton-exe

# 5. Леволинейная без пустых правил
echo "S=Aa|b;A=c" | stack exec regexp-to-automaton-exe

# 6. Леволинейная только с терминалами
echo "S=a|b" | stack exec regexp-to-automaton-exe

# 7. Ошибка: неопределённый нетерминал
echo "H=aA;A=bB" | stack exec regexp-to-automaton-exe

# 8. Ошибка: смешанные типы
echo "H=aA|Ba" | stack exec regexp-to-automaton-exe

# 9. Ошибка: пустое правило в праволинейной
echo "H=aA;A=" | stack exec regexp-to-automaton-exe

# 10. Грамматика с пробелами и лишними `;`
echo " H = a A | b ; A = c ; " | stack exec regexp-to-automaton-exe
```

## Структура проекта

| Модуль          | Назначение                                            |
|-----------------|-------------------------------------------------------|
| `Grammar.hs`    | Парсинг, определение типа и валидация грамматики      |
| `Automaton.hs`  | Типы данных НКА/ДКА и функции преобразования в рёбра  |
| `Convert.hs`    | Построение НКА по грамматике                          |
| `Determinize.hs`| Проверка детерминированности и детерминизация         |
| `app/Main.hs`   | Точка входа и форматированный вывод                   |
